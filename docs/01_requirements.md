文書名: 研究発表WEBプラットフォーム MVP 要件定義書
バージョン: v1.0
作成目的: 3日以内で動作するMVPの開発範囲と要件を定義するため
対象プロジェクト: 研究発表がWEB上でできる会員制プラットフォーム
想定技術: Next.js App Router / Supabase / Vercel

1. 背景

従来の研究発表運用では、発表エントリー、原稿提出、事務局確認、修正依頼、再提出、公開、公開後の議論が、メール、Googleフォーム、チャットツール、ファイル共有サービスなどに分散しやすい。

その結果として、以下の課題が生じる。

・投稿状況が一覧で把握しづらい
・誰が何をいつ対応すべきか分かりづらい
・修正依頼と再提出の履歴が追いづらい
・公開タイミングの管理が煩雑
・会員限定公開と一般公開の切替が面倒
・発表ごとの議論が分散する

これらの課題を解決するため、本プロジェクトでは、研究発表に関する一連の流れを単一のWEBプラットフォーム上で完結できる仕組みを構築する。

2. 本システムの目的

本システムの目的は、研究発表者、事務局、閲覧者の3者が、研究発表の投稿から公開、議論までを一つのサイト上で完結できるようにすることである。

MVP段階では、以下を実現する。

・研究発表者がWEB上で発表エントリーできる
・研究発表者がWEB上で原稿を提出できる
・事務局が原稿を確認し、修正依頼を出せる
・研究発表者が修正版を再提出できる
・指定日に原稿を会員限定で公開できる
・発表ごとにコメントスレッドを用意できる
・一定期間だけ議論可能にできる
・後日、フリーアクセス可能な状態へ切り替えられる

3. スコープ

3-1. 対象範囲

本MVPで対象とする範囲は以下とする。

・会員制の研究発表公開サイト
・研究発表者による投稿および再提出
・事務局による確認および修正依頼
・会員限定での論文公開
・論文ごとのコメントスレッド
・議論期間の制御
・後日の一般公開切替
・基本的な管理画面
・認証、データ保存、ファイル保存、公開制御

3-2. 非対象範囲

本MVPでは以下は対象外とする。

・匿名査読機能
・複数レビュアーの割当管理
・採点式レビュー
・SlackやTeamsとの正式連携
・通知メールの高度自動化
・リアルタイム共同編集
・LaTeXコンパイル基盤
・DOI連携
・多言語化
・決済機能
・高度な分析ダッシュボード
・学会運営全般機能
・動画配信やライブ発表機能

・投稿詳細確認
・修正依頼

役割

・ログイン

・論文詳細閲覧
・コメント投稿
・コメント閲覧

5. 提供価値

本システムが提供する価値は以下の通り。

研究発表者向け
・投稿から再提出までをサイト内で完結できる
・今の状況が分かりやすい
・修正指示内容が一元管理される
・公開後の議論にも同一サイトで参加できる

事務局向け
・投稿状況が一覧で管理できる
・修正依頼と再提出の履歴を追える
・公開日時や公開範囲を管理できる
・議論期間を制御できる

閲覧者向け
・会員限定の研究発表を安心して閲覧できる
・発表単位で議論に参加できる
・議論可能期間が明確で分かりやすい

6. システムコンセプト

本システムは、重厚な査読管理システムではなく、研究発表運用をWEB化する軽量な投稿管理プラットフォームとして設計する。

コンセプトは以下とする。

・研究発表運用を一元化する
・研究者が直感的に使える
・事務局の管理負荷を下げる
・会員限定公開と一般公開切替を簡単にする
・3日で動くMVPとして成立させる
・将来の機能拡張を見据えた構造にする

7. 成功条件

MVPの成功条件は以下とする。

・研究発表者が会員登録し、発表エントリーできる
・研究発表者が原稿ファイルを提出できる
・事務局が投稿内容を確認し、修正依頼を出せる
・研究発表者が修正版を再提出できる
・事務局が公開日時を設定できる
・公開後、会員だけが論文を閲覧できる
・論文ごとにコメントスレッドを持てる
・議論期間終了後、新規コメントを停止できる
・事務局が後日フリー公開へ切り替えられる
・最低限の管理画面が動作する
・Vercel上で本番動作可能な状態になる

8. MVP前提条件

・開発期間は3日
・デザインは過剰に作り込まず、UIはシンプルかつ明快にする
・実装優先順位は、見た目よりワークフロー成立を優先する
・リアルタイム性より運用の安定性を優先する
・会員限定公開は authenticated user 前提で制御する
・事務局権限は初期段階では限定人数の手動付与で運用する

9. 業務フロー要件

9-1. 発表エントリー

研究発表者は会員登録後、発表エントリーを作成できること。
エントリー時に最低限入力する情報は以下とする。

・発表タイトル
・概要
・カテゴリまたは分野
・著者名
・所属
・発表者情報
・公開希望日
・議論希望期間

9-2. 原稿提出

研究発表者はエントリー済みの発表に対して、原稿ファイルを提出できること。
MVPではPDF提出を基本とする。

提出時に必要な情報
・原稿ファイル
・任意メモ
・提出日時

9-3. 事務局確認

事務局は提出済み原稿を一覧から確認できること。
事務局は以下のいずれかを選択できること。

・差し戻し
・公開準備へ進める

差し戻し時は、修正指示コメントを記録できること。

9-4. 再提出

研究発表者は差し戻し済み原稿に対して修正版を再提出できること。
再提出後は履歴が保持されること。

9-5. 会員限定公開

事務局は規定日に発表原稿を会員限定で公開できること。
公開された論文は、ログイン済み会員のみ閲覧可能であること。

9-6. コメントスレッド

各論文には専用のコメントスレッドを持てること。
議論期間中は、研究発表者および閲覧者がコメント投稿できること。
議論期間終了後は、閲覧は可能だが新規投稿は不可とすること。

9-7. フリー公開

事務局は後日、会員限定公開中の論文を一般公開状態へ切り替えられること。
一般公開後は、非ログインユーザーでも閲覧可能とする設計余地を持つこと。
ただしMVP時点では、公開制御の実装方法は運用しやすさを優先する。

10-2. 研究発表者向け機能

・マイページ表示
・自分の投稿一覧表示
・新規発表エントリー作成
・発表情報編集
・原稿提出
・再提出
・修正指示の確認
・公開状態の確認
・コメント投稿

10-3. 事務局向け機能

・投稿一覧表示
・ステータス別絞り込み
・投稿詳細確認
・修正依頼
・公開日設定
・公開切替
・議論期間設定
・コメント監視または削除
・簡易ユーザー管理

10-4. 閲覧者向け機能

・論文一覧表示
・論文詳細表示
・公開中論文の検索または絞り込み
・コメント閲覧
・コメント投稿

10-5. コメント機能

・論文ごとのスレッド表示
・コメント投稿
・コメント一覧表示
・議論期間外の投稿禁止
・必要最低限の削除対応

10-6. ファイル管理機能

・PDFアップロード
・提出版ごとの保存
・事務局と発表者が必要に応じて参照可能
・公開時に閲覧可能状態へ制御

11. 非機能要件

11-1. 開発効率

・3日以内で実装可能な構成であること
・Next.js App Router と Supabase を中心に構成し、余計な中間層を持たないこと
・初期実装で複雑なジョブ基盤を持たないこと

11-2. 操作性

・研究発表者が迷いにくい画面構成であること
・事務局がステータスを見て即判断できること
・閲覧者が公開状況と議論状態を理解しやすいこと

11-3. 保守性

・データ構造が将来拡張に耐えられること
・提出履歴を残せること
・ロール制御を後から拡張しやすいこと

11-4. セキュリティ

・会員限定公開資料は未ログイン状態で閲覧できないこと
・権限外ユーザーが投稿管理画面へ入れないこと
・原稿ファイルの参照制御が行えること
・RLSを前提に設計すること

11-5. 性能

・MVP時点では小規模運用を想定する
・論文一覧、詳細、コメント一覧が通常利用に耐える速度で表示できること
・大規模負荷対応は今後の拡張で考える

14. ステータス要件

論文または投稿の状態は以下を基本とする。

・draft
・submitted
・revision_requested
・resubmitted
・scheduled
・published_members_only
・free_public
・archived

必要要件
・各状態が明確に定義されること
・事務局のみが変更できる状態を明示すること
・発表者が変更できる範囲を明示すること
・画面上で現在状態が分かること

15. 権限要件

研究発表者
・自分の投稿の作成、編集、提出、再提出が可能
・自分の投稿の修正指示を確認可能
・公開後に自分の論文へコメント可能

事務局
・全投稿の閲覧が可能
・投稿状態変更が可能
・修正依頼が可能
・公開設定が可能
・スレッド管理が可能
・コメント削除または制御が可能

閲覧者
・公開中の会員限定論文の閲覧が可能
・議論期間中のコメント投稿が可能
・管理機能は不可

フロントエンド

バックエンド基盤
・Supabase Auth

開発方針
・Route Handler または Server Actionsで実装

17. 制約条件

・開発期間は3日
・MVPであるため機能は厳選する
・管理者機能は簡易実装を前提とする
・UIは実用性優先
・運用で吸収できる部分は一部運用で補う
・高度な通知やレビュー機能は後続フェーズへ送る

18. 想定リスク

・投稿ワークフローを欲張ると3日では破綻する
・匿名査読まで入れると設計が重くなる
・公開制御とファイル制御を雑にすると会員限定要件が崩れる
・コメント機能を作り込みすぎると本質から外れる
・管理画面の作り込みすぎはMVPの妨げになる

19. リスク対応方針

・査読ではなく発表運用システムとして割り切る
・PDF提出を前提にして入力自由度を下げる
・ロールは3種に限定する
・公開状態は最小限のステータスに絞る
・コメントはシンプルなスレッド方式にする
・リアルタイム更新は後回しにする

21. 受け入れ条件

以下を満たした場合、本MVPは受け入れ可能と判断する。

・会員登録とログインができる
・研究発表者が発表エントリーできる
・PDF原稿を提出できる
・事務局が差し戻しコメントを登録できる
・研究発表者が再提出できる
・事務局が公開日時を設定できる
・会員限定で論文を閲覧できる
・論文ごとにコメントスレッドを表示できる
・議論期間外ではコメント投稿できない
・後日、一般公開へ切替可能な設計になっている
・最低限の管理画面で運用できる

22. 要件定義上の最終判断

本MVPは、研究発表に必要な投稿、確認、修正、公開、議論という中核フローを成立させることを最優先とする。

本段階では、研究者にとって使いやすく、事務局にとって運用しやすく、3日で形になることを最重要要件とする。

したがって、本要件定義では以下を最終方針とする。

・機能は投稿運用に集中する
・査読の高度化は行わない
・会員限定公開と議論機能をMVPの中心価値とする
・将来拡張可能な構造を持たせる
・まずは確実に動くプロトタイプを成立させる
